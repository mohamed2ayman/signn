- Backend module: `backend/src/modules/contract-sharing/` (3 files, present on disk).

## C. Auth & access model

### C1. JWT strategy

- File: `backend/src/modules/auth/strategies/jwt.strategy.ts`.
- `JwtStrategy.validate()` (lines 39-72) expects payload claims `sub` (user ID), `email`, `role`, `organization_id`, optional `jti` (Phase 4.2), `exp`, `iat`.
- Validation order: user exists + active → if `jti` present, Redis blacklist check (fails-OPEN on Redis error) → return `{ id, email, role, job_title, organization_id }`.
- Pre-Phase-4.2 tokens (no `jti`) get a warning but pass under a 7-day grace window — that window will have expired in production by the time this audit is acted on; treating it as effectively closed is safe.

### C2. Guards in use

- `JwtAuthGuard` — `backend/src/modules/common/guards/jwt-auth.guard.ts` (5 lines, thin `AuthGuard('jwt')` wrapper). Applied **per-controller / per-method**, never globally.
- `RolesGuard` — `backend/src/modules/common/guards/roles.guard.ts` (28 lines, reads `@Roles()` metadata).
- `PermissionLevelGuard` — `backend/src/common/guards/permission-level.guard.ts`. Reads `@RequirePermission(PermissionLevel.X)` metadata; checks the user's `ProjectMember.permission_level` against the required floor. Bypasses for `OWNER_ADMIN` and `OPERATIONS` (lines 61-68, per audit-D1 reading).
- `SubscriptionGuard` — `backend/src/common/guards/subscription.guard.ts` exists on disk (confirmed by `find`); body not opened in this audit. (uncertain: details.)
- `ThrottlerGuard` — applied **only** via `@ThrottleOnly('bucket-name')` (`backend/src/common/decorators/throttle-only.decorator.ts`). The composed helper sets `@UseGuards(ThrottlerGuard) + @Throttle + @SkipThrottle` so a method obeys only its own bucket (per CLAUDE.md Rate Limiting Policy rule 1). `THROTTLER_NAMES` const lists `['login','register','forgot','reset','mfa','recovery','refresh','invitation','waitlist','portfolio_export']`.
- **There is no `@Public` / `@SkipAuth` / `@AllowAnonymous` decorator in the codebase.** Verified by grep — zero matches across `backend/src/`. The convention for "no-auth endpoint" is **omission**: simply do not stack `@UseGuards(JwtAuthGuard)` on the method. This is how the contract-sharing public read, the public-parties controller, the obligation mark-as-met controller, the portfolio-export download controller, and the DocuSign webhook all work.
- **No global `APP_GUARD` provider.** Verified by reading `app.module.ts` providers + `main.ts`. Auth is opt-in per-controller / per-method.

- Invitation creation site: an org-invite endpoint should exist under organizations/users module — not opened in this audit; inferred from CLAUDE.md references to `invitations` table + `auditLog` action `invitation_sent` (uncertain: exact endpoint path).

### C5. Project-party invitation flow (guest, NO account)

Parallel and distinct from C4.
- Public controller: `backend/src/modules/project-parties/public-parties.controller.ts` — `GET /api/v1/public/parties/invitation/:token` + `POST /api/v1/public/parties/invitation/accept`.

### C6. Per-role obligation permission model (Phase 7.15)

- **No `canEdit` / `canDelete` / `canAssign` / `isOwnerOrAssignee` service-layer helper was found** by grep across `backend/src/modules/obligations/` and `backend/src/modules/compliance/`. (inference based on absence of matches.)
- The dominant pattern is **controller-layer** gating: `@UseGuards(JwtAuthGuard, RolesGuard, PermissionLevelGuard) + @RequirePermission(PermissionLevel.X)` (example: claims.controller.ts line 28, 33, 43, 65, 75; same pattern on notices.controller.ts line 27). Service methods then trust the controller has gated, and only enforce **org-scoping** (e.g. `assertContractInOrg(contractId, orgId)` in `backend/src/modules/negotiation/negotiation.service.ts:46`).
- **What this means for a guest-permission model:** there is **no precedent service-layer pattern to mirror**. CLAUDE.md's "Phase 7.15 per-role obligation permission model" did not (in the files read) leave a service-layer permission helper a guest-portal could compose against. The architecture session should treat per-role permissions at the service layer as a NEW pattern, not an extension. (medium confidence — grep is not proof of absence.)

### C7. No-account / tokenised-access precedents (the "magic link" lineage)

Three live token systems exist; each has different security/UX trade-offs the architecture session should know:

| Endpoint | Token shape | TTL | Reusable? | Revocation | HMAC-before-DB invariant? | Notes |
|---|---|---|---|---|---|---|
| `GET /api/v1/public/obligations/mark-met?token=...` | base64url(`{payload_json}.{hmac_sha256}`) | 7 days (hardcoded default in `obligation-token.service.ts:24`) | Single-use (nonce on obligation row, cleared on first use) | Burn-on-use via nonce | Yes (line 52, `timingSafeEqual` before DB read) | Single-action token (mark a specific obligation met) |
| `GET /api/v1/portfolio-exports/download?token=...` | `<b64url(payload)>.<b64url(hmac_sha256(b64_payload))>` | Caller-supplied `expiresAt` Date; per Phase 7.17 spec = 1 hour | Reusable inside TTL | **No revoke mechanism** — only expiry | Yes (lines 98-108, regression-tested) | Single-resource token (download a specific PDF job's file); bare-HTTP, no `@UseGuards` |
| `GET /contract-sharing/shared/:token` | Opaque `crypto.randomBytes(32).toString('hex')` stored verbatim in DB | Optional, set by sharer (1d / 7d / 30d / 90d / Never) | Reusable | Yes — `is_active=false` set by sharer via `DELETE /contract-sharing/:id` | **No HMAC** — token authority is DB row presence + `is_active=true` + `expires_at` check. Lookup is by raw token (single DB hit) | Multi-resource read (contract + clauses + risks) — closer in shape to what a Guest Portal session would need than the other two |
| `POST /docusign/webhook` | DocuSign HMAC-SHA256 of raw body, header `X-DocuSign-Signature-1` | n/a | n/a (event-stream) | n/a | Yes (lines 61-72) | Inbound webhook, not a user-facing magic link |

The contract-sharing token is the closest pre-existing pattern to a "guest session token" — but its current security floor is **lower** than the HMAC-signed precedents and it has **no rate-limiting** decorator on the public-read endpoint.

## D. The four guest capabilities — equal-depth map

### D1. View a single assigned contract

**Backend (EXISTS):**

**Frontend (EXISTS):**
- `apps/sign/src/pages/app/ContractDetailPage.tsx` — at route `/app/contracts/:id` (App.tsx line 237), wrapped in `ProtectedRoute` with `APP_ACCESS_ROLES` (i.e. OWNER_* + SYSTEM_ADMIN + OPERATIONS).
- 9 tabs exposed (lines ~237-255 per audit): Clauses, Risk Analysis, Obligations, Claims, Notices, Sub-Contracts, Compliance, History, Approvals.

**Wired vs stubbed:**
- `GET /contracts/:id` and `ContractDetailPage` are fully WIRED end-to-end for OWNER_* users.
- **Org-scoping caveat:** the auditor reading `findById` reported the lookup is by `{ id }` only — no `WHERE project.organization_id = :orgId` filter visible in the service method body. If true, this is a pre-existing tenancy gap (a user with a valid JWT for org X who knows a contract ID from org Y could read it). **Uncertain** — the service may rely on the controller-side `PermissionLevelGuard` querying `ProjectMember` to keep things scoped, which would prevent the cross-org read in practice. Architecture session should re-verify before assuming either way.

**Blockers for a no-account guest:**
1. `JwtAuthGuard` on `GET /contracts/:id` — guest has no JWT.
2. `PermissionLevelGuard` on the same controller — guest has no `ProjectMember` row, so even with a JWT they would fail unless OWNER_ADMIN / OPERATIONS (which bypass per lines 61-68).
3. `/contractor/contracts/:id` route does not exist in the router — confirmed by reading `apps/sign/src/App.tsx` lines 294-305.
4. The **contract-sharing public endpoint** (`GET /contract-sharing/shared/:token`) IS a no-account path to view a contract — but no frontend route renders its response (Area E).

## G. Conventions to mirror

### G1. NestJS module wiring

- Canonical recent example: `backend/src/modules/portfolio-export/` (Phase 7.17 2c).
- Sub-folder convention: `controllers/`, `services/`, `processors/` (BullMQ jobs), `schedulers/` (cron), `dto/`, `entities/`, `constants/`.
- Module registration: in the module's own `.module.ts`, imports `TypeOrmModule.forFeature([Entity])` + `BullModule.registerQueue({ name: ... })` + other modules whose services are injected.
- App-level wiring: append the module to the `imports[]` array in `backend/src/app.module.ts` (in a logically-grouped position).
- Simpler reference: `backend/src/modules/waitlist/` (Phase 6.9) — flatter layout, single controller/service.

### G2. React route + page + components

- Router: `apps/sign/src/App.tsx`. Routes registered as JSX `<Route>` elements grouped by portal (`/app/*`, `/admin/*`, `/contractor/*`, public auth routes).
- Each portal wraps its children in a `ProtectedRoute` + layout shell:
  - `/app/*` → `AppLayout` with `clientNavItems` (App.tsx ~213-256).
  - `/admin/*` → `AdminLayout` with its own internal nav (~258-292).
  - `/contractor/*` → `AppLayout` with `contractorNavItems` (~294-305) — currently 1 child route only.
- Page directories: `apps/sign/src/pages/app/` (29 pages), `pages/admin/` (21), `pages/auth/` (8), `pages/contractor/` (2 — stub dashboard + accept-invitation).
- Mobile shell: `AppLayout` is already mobile-responsive after Phase 6.4 (canonical 768px breakpoint, compound `md:ltr:` / `md:rtl:` variants for compound RTL+responsive overrides — see CLAUDE.md Phase 6.4 hard rule 2).
- Fine-grained role gating composes via nested `ProtectedRoute` with `allowedRoles=[UserRole.X]` (App.tsx line 227 — portfolio gated to OWNER_ADMIN inside the broader APP_ACCESS_ROLES).

### G3. i18n keys + RTL

- Locale files: `apps/sign/src/i18n/locales/{en,ar,fr}/common.json` — all 3 present. EN ≈ 687 lines, AR ≈ 679, FR ≈ 695.
- Init: `apps/sign/src/i18n/index.ts:1-49`. Imports all 3 locale JSONs; `fallbackLng: 'en'`; on `languageChanged`, sets `document.documentElement.dir = 'rtl' | 'ltr'` (lines 43-46).
- Language switcher: `apps/sign/src/components/common/LanguageToggle.tsx`. `selectLanguage(code)` calls `i18n.changeLanguage` + updates localStorage `i18nextLng` + sets `documentElement.dir` and `.lang`.
- Namespace convention by area: `auth.*`, `nav.*`, `portal.*`, `userType.*`, `common.*`, plus feature-namespaces (`portfolio.*`, `obligation.*`, `contract.tabs.*`).
- **Hard rule** (CLAUDE.md): every new key must be added to all 3 locales in the same commit. Phase 7.1 Step 2 set the precedent — 53 keys × 3 locales in one commit.
- RTL clause display rule: every render of `obligation.description`, `obligation.timeframe_description`, clause titles + content must carry `dir="auto"` + `style={{ unicodeBidi: 'plaintext' }}` (CLAUDE.md, Arabic Text Display rule + Phase 7.1 Step 2 hard rule 2).

### G4. Rate-limiting

- Decorator: `backend/src/common/decorators/throttle-only.decorator.ts`. Constant `THROTTLER_NAMES` is the canonical list — currently 10 buckets including `'invitation'`, `'waitlist'`, `'portfolio_export'`.
- Hard rule: **always** use `@ThrottleOnly('bucket-name')`, never plain `@Throttle({...})` (Phase 6.9 hard rule 2 + Rate Limiting Policy hard rule 1).
- App registration: `backend/src/app.module.ts:173-203` registers all buckets via `ThrottlerModule.forRootAsync()` with Redis storage; `app.set('trust proxy', 1)` is required in `main.ts` for IP-based throttling to key off the real client IP.

### G5. Input sanitization

- `backend/src/common/utils/escape-like.ts` — `escapeLikeParam(value)` for ILIKE search; always wrap the call site with the `%` outside.
- `backend/src/common/utils/sanitize.ts` — `stripHtml(value)` via `@Transform` decorator on every free-text DTO field that should not contain HTML.
- `backend/src/common/utils/escape-html-email.ts` — `escapeHtml(value)` for every user-supplied string interpolated into an email template (Phase 3.5 — 38 call sites in `templates/index.ts`).
- DTO pattern: stack `@IsString()` + `@MaxLength(N)` + `@Transform(({value}) => stripHtml(value))`. `@MaxLength` MUST be on both create AND update DTOs (Phase 3.2 hard rule). Canonical example: `backend/src/modules/contracts/dto/add-comment.dto.ts`.

### G6. Migration convention

- Path: `backend/src/database/migrations/`.
- Naming: `<epoch-ms>-<DescriptiveName>.ts` (e.g. `1750000000001-AddContractValueAndCurrency.ts`).
- **Hard rule (Phase 7.9, lesson #111):** never use `EXCEPTION WHEN ... THEN null` — silent migration failures are the documented anti-pattern that masked the Phase 7.3 obligation-status enum bug for months.
- For CREATE TYPE: wrap in `DO $$ BEGIN IF NOT EXISTS (SELECT 1 FROM pg_type WHERE typname = '...') THEN CREATE TYPE ... ; END IF; END $$;`.
- For ADD CONSTRAINT: `IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname = '...')`.
- For ALTER TYPE ADD VALUE: use `IF NOT EXISTS` clause directly on the ALTER, AND set `transaction = false` on the migration class (PostgreSQL forbids ALTER TYPE ADD VALUE inside a transaction block, AND `migrationsTransactionMode: 'each'` is set in `backend/src/config/data-source.ts:40` to allow per-migration overrides).
- `data-source.ts` runs outside Nest bootstrap and must validate `DATABASE_URL` itself (lines 9-26) — Joi doesn't fire there.

### G7. Startup schema-assertion (enum safety)

- Pattern: a service implementing `OnModuleInit` that queries `pg_enum` at bootstrap and throws if any required value is missing.
- Canonical: `backend/src/modules/obligations/obligation-schema-check.service.ts:1-72`. Implements `OnModuleInit`; queries `SELECT enumlabel FROM pg_enum WHERE enumtypid = 'obligation_status'::regtype ORDER BY enumsortorder`; verifies `REQUIRED_VALUES = ['MET', 'WAIVED']` are present; throws with a developer-friendly error otherwise.
- Wired in: `backend/src/modules/obligations/obligations.module.ts:39` (added to `providers[]`).
- This pattern was instituted in direct response to Phase 7.3 — when an enum is mark-as-met-style (i.e. live code depends on specific enum values existing), this check is the protection layer against a silent migration failure recurring.

(high confidence — confirmed by reading both the controller method and the service method bodies; the only mitigating factor that could prevent the gap would be middleware/interceptor outside the guard stack, and none were observed.)

### K.5 — Summary

Static analysis (K.1) ruled out every plausible mitigation outside the per-controller guard stack. Live runtime probes (K.3) confirmed the predicted behavior end-to-end against the running backend: **any authenticated user, of any role, in any organization, can read the full contents of any contract in any other organization by hitting `GET /api/v1/contracts/:id` with a valid JWT and the contract's UUID.** The response includes the contract creator's bcrypt password hash via the loaded `creator` relation — a second, distinct vulnerability surfaced by the same endpoint.

J.2's prediction is confirmed. Whether to address this gap, when, and how, is for the architecture session — out of audit scope. No source files were modified during this verification.
