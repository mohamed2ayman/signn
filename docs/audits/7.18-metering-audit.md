## 2. Free-vs-metered classification

### 2.1 Generative-on-demand (would be metered)

Every endpoint in the table at §1.3 that dispatches to a Celery agent task burns
Claude tokens during the request. There is **no caching layer in front of any of
them**. The Bull-queue chain inside document-processing means a single contract
upload triggers FOUR generative tasks (text extract is free OCR; clauses, risk,
obligations, conflicts are all generative) automatically.

### 2.2 Serves-precomputed (free)

| Surface | Why free | Citation |
|---|---|---|
| `GET /api/v1/ai/jobs/:jobId` polling | Reads Celery result backend (Redis), no AI call | ai.controller.ts:105-108 |
| `GET /api/v1/contracts/:id/clauses` and clause-detail reads | Returns rows from `clauses` table (precomputed by `extract-clauses`) | contracts/clauses modules |
| `GET /api/v1/contracts/:id/risk-analyses` and explanation reads | Returns rows from `risk_analyses` (precomputed) | risk-analysis module |
| `GET /api/v1/contracts/:id/obligations` (+ portfolio + calendar) | Returns rows from `obligations` (precomputed) | obligations + compliance modules |
| `GET /api/v1/contracts/:id/compliance-checks` and finding reads | Returns rows from `compliance_findings` | compliance module |
| `GET /api/v1/chat/sessions/:id/messages` history | Returns rows from `chat_messages` | chat module |
| OCR `POST /agents/extract-text` (in isolation) | Tesseract or Textract, no LLM tokens | tesseract_text_extractor.py |

**This is the spine of the metering distinction.** Anything that reads a previously
written row stays free. The metered moment is the **write** (the AI call that
produced the row). The viewer credential, which serves the precomputed AI clause
classification on the bound contract, is explicitly framed as zero-new-cost — confirmed
verbatim at [viewer-credential.service.ts:14-15](backend/src/modules/guest-portal/services/viewer-credential.service.ts):
"the existing AI clause classification — zero new cost".

### 2.3 Hybrids (riskiest to meter — name each one)

These are call sites where the same NestJS service either reads a cached row OR
regenerates depending on conditions; they are the ones where a `force=true`
parameter, a status flag, or a missing row silently triggers a new metered AI call.

1. **`compliance.service.runCheck()` (≈ line 135).** Creating a new ComplianceCheck row
   ALWAYS dispatches the agent — it is NOT idempotent on "same contract checked
   recently." A caller that resubmits gets a new check + new tokens.

2. **Obligation extraction inside compliance pipeline (compliance.service.ts ≈ line 287).**
   The compliance pipeline polls and, if obligations are missing, re-triggers
   `triggerExtractObligations`. So an org that re-runs compliance can trigger
   obligation re-extraction transparently.

3. **`document-processing.service.reprocess()`.** Explicitly clears prior clauses
   (lesson #9 in CLAUDE.md) and re-dispatches the full extract/risk/obligations/
   conflicts chain — 4 generative tasks per call.

4. **`document-processing.service.queueRiskAnalysis` / `queueObligationsExtraction` /
   `queueConflictDetection`.** Auto-fire after clause extraction completes; there is no
   "skip if already analysed" gate visible in the wiring. The 7.18 Guest Portal audit
   ruled out a pre-existing cache; the upload→4-job fan-out is the default.

5. **Embedding search.** Calls OpenAI **once per query** to generate a search vector,
   then does local cosine similarity. Reads (the vector store lookup) are free, but
   every search incurs one embedding call.

6. **Chat (multi-turn).** Each message in a session is a fresh, unbounded Claude
   call with the full session history packed in as context — token cost grows with
   conversation length, with no built-in cap.

**These six are the highest-risk wiring points.** Anything that wraps the metered call
must not be bypassable by hitting one of these paths.

## 3. AI-backend cost surface & usage reporting

### 3.1 Response shapes from the ai-backend

[ai-backend/app/routers/agents.py:40-268](ai-backend/app/routers/agents.py) — every
dispatch endpoint returns:

```python
return {"job_id": str(task.id), "status": "queued"}
```

[ai-backend/app/routers/agents.py:230-247](ai-backend/app/routers/agents.py) — the
job-status endpoint returns:

```python
return {
    "job_id": job_id,
    "status": result.status,        # "pending" | "completed" | "failed"
    "result": result.result,        # the task's return dict on completion
}
```

[ai-backend/app/tasks.py:32-204](ai-backend/app/tasks.py) — every task returns a dict
of shape `{"status": "completed" | "failed", "result": ...}` (or `"error": ...` on
failure). **No task returns a `usage` / `tokens` / `cost` field.**

[ai-backend/app/routers/embeddings.py:24-35](ai-backend/app/routers/embeddings.py) —
ingest returns `{status, asset_id}`; search returns `{results}`. **No usage field.**

### 3.2 The Anthropic / OpenAI SDK responses DO carry usage — but the code discards it

Verified by direct grep across `ai-backend/app/agents/`:

```bash
grep -rn "\.usage\|input_tokens\|output_tokens\|token_count\|TokenUsage" ai-backend/app
```

returns **zero matches**. Every one of the 10 agents calls `self._client.messages.create(...)`,
reads `message.content[0].text`, and discards the `message.usage` field that the
Anthropic SDK puts on the response object. Same story for the embedding service —
`response.data[0].embedding` is read; `response.usage.prompt_tokens` (which OpenAI
returns) is dropped. Sample sites:

- [ai-backend/app/agents/risk_analyzer.py:186-194](ai-backend/app/agents/risk_analyzer.py)
- [ai-backend/app/agents/clause_extractor.py:569-574](ai-backend/app/agents/clause_extractor.py)
- [ai-backend/app/agents/summarizer.py:64-80](ai-backend/app/agents/summarizer.py)
- [ai-backend/app/agents/conversational_agent.py:112+](ai-backend/app/agents/conversational_agent.py)
- [ai-backend/app/services/embedding_service.py:34-40](ai-backend/app/services/embedding_service.py)

### 3.3 Models / providers

| Agent (ai-backend) | Provider | Model | Source |
|---|---|---|---|
| All 9 LLM agents | Anthropic | `claude-sonnet-4-6` (hardcoded in every agent) | risk_analyzer.py:187, clause_extractor.py:570, summarizer.py:65, etc. |
| Embeddings (ingest + search) | OpenAI | `text-embedding-3-small` | embedding_service.py |
| Text extraction (OCR, no LLM) | Internal | Tesseract (factory pattern w/ deferred Textract) | tesseract_text_extractor.py |

`claude-sonnet-4-6` is the hardcoded model identifier in every agent file. There is no
config option to override per-org. The Anthropic API key lives in the ai-backend env
(`ANTHROPIC_API_KEY`); NestJS never sees it.

### 3.4 Existing token-accounting / usage-logging / cost-tracking infrastructure

**None.** Grep across `backend/src` AND `ai-backend/app`:

- No `ai_usage` / `ai_tokens` / `ai_cost` table in the entity tree
- No `usage_log` / `token_log` service
- `backend/src/modules/admin-billing/` exists but tracks payment transactions
  (Paymob), NOT per-call AI usage
- `knowledge_asset_usages` (Phase 7.24b) IS a backlink table mapping (asset →
  contexts that used it), NOT a metering ledger. Different concept; not a precedent.

### 3.5 What this means for reconcile

The reconcile step of reserve→run→reconcile can record only a **run-count** today, NOT
**true token cost**. To record true cost, the ai-backend agents (all 9 + embeddings)
must be modified to capture `message.usage.input_tokens` + `message.usage.output_tokens`
and return them on the Celery task result. NestJS must then read those off
`getJobStatus().result.usage` and feed them into the reconcile. This is a non-trivial
ai-backend change that the metering build's scope must explicitly decide on.

The locked design names three caller shapes. What today's code says about each:

   [user.entity.ts](backend/src/database/entities/user.entity.ts) lines 99-103 define `AccountType` enum: `MANAGING | GUEST | FREE`.

2. **Guest user row** — `User` row with `role=GUEST`, `account_type=GUEST`,

   `establishIdentity` transaction at
   [guest-invitation.service.ts:231-365](backend/src/modules/guest-portal/services/guest-invitation.service.ts).
   Returns a standard JWT (access_token + refresh_token); identity-promoted guest
   then authenticates **exactly like a managing user** with `account_type=GUEST`

## 5. Existing metering / rate-limit / quota / throttle infra

### 5.1 NestJS Throttler (Redis-backed, per-IP)

[backend/src/app.module.ts:196-234](backend/src/app.module.ts) configures 11 named
throttlers, all backed by `ThrottlerStorageRedisService` against the same `REDIS_URL`
used by Bull and the token blacklist:

| Bucket name | TTL | Limit | Applied at |
|---|---|---|---|
| `login` | 600_000 ms | 5 | `POST /auth/login` |
| `register` | 3_600_000 | 3 | `POST /auth/register` |
| `forgot` | 3_600_000 | 3 | `POST /auth/forgot-password` |
| `reset` | 900_000 | 5 | `POST /auth/reset-password` |
| `mfa` | 600_000 | 5 | `POST /auth/verify-mfa` |
| `recovery` | 3_600_000 | 3 | `POST /auth/verify-recovery` |
| `refresh` | 900_000 | 20 | `POST /auth/refresh` |
| `invitation` | 3_600_000 | 5 | `POST /auth/accept-invitation` |
| `waitlist` | 3_600_000 | 3 | `POST /waitlist` |
| `portfolio_export` | 900_000 | 5 | `POST /portfolio-exports` (Phase 7.17 2c) |
| `guest_invite_exchange` | 900_000 | 10 | `POST /public/guest-invitations/exchange` (Phase 7.18) |

All are **per-IP only**, never per-org, never per-user. Decorator pattern:
[throttle-only.decorator.ts](backend/src/common/decorators/throttle-only.decorator.ts)
exports `THROTTLER_NAMES` const + `@ThrottleOnly(name)` composer that disables every
other bucket on the endpoint via `@SkipThrottle`. The hard rule in CLAUDE.md (Rate
Limiting Policy section) forbids plain `@Throttle` — verified clean in the codebase.

**This substrate is per-IP rate-control. It is not a metering substrate.** No bucket is
keyed on `(org_id, action)`. No bucket accumulates against a plan-driven cap. Eleven
buckets cover unauthenticated endpoints; **zero buckets cover authenticated AI
endpoints**.

### 5.2 Redis-backed counters outside the throttler

Only one: **TokenBlacklistService**
([backend/src/common/services/token-blacklist.service.ts](backend/src/common/services/token-blacklist.service.ts)).
Key shape `blacklist:jti:{jti}`, TTL set to access-token remaining lifetime, fails-OPEN
on Redis error. Used by `JwtStrategy.validate()` to enforce logout/family-revocation
across access tokens.

Grep for `new Redis(`, `ioredis`, `rate:`, `quota:`, `count:`, `limit:` keys outside
Bull and the throttler returns no other Redis counter patterns.

### 5.3 Plan-driven enforcement that already exists (defined, never wired)

[backend/src/common/guards/subscription.guard.ts](backend/src/common/guards/subscription.guard.ts)
is a fully-implemented NestJS guard that:

- Loads the org's `OrganizationSubscription` where `status=ACTIVE`
- Throws `ForbiddenException` with code `NO_SUBSCRIPTION` if there's no active row
- Throws if `end_date < now`
- Compares current resource counts against
  `plan.max_projects` / `plan.max_users` / `plan.max_contracts_per_project`
- Throws `PLAN_LIMIT_PROJECTS` / `PLAN_LIMIT_USERS` / `PLAN_LIMIT_CONTRACTS` on
  exceedance

**It is attached to ZERO endpoints.** Verified directly by grep:

```bash
grep -rn "SUBSCRIPTION_CHECK_KEY" backend/src --include="*.ts"
```

returns only the guard's own file. Zero `@SetMetadata(SUBSCRIPTION_CHECK_KEY, ...)`
sites, zero `@CheckSubscription(...)` decorator helpers, zero `@UseGuards(...SubscriptionGuard...)`
in any controller. **A complete plan-limit enforcement substrate exists but is dead
code today.**

This matters: the metering build either (a) plumbs `meter_key`-based caps through
`SubscriptionGuard` (extending it) and quietly turns it on for the legacy
`max_projects`/`max_users`/`max_contracts_per_project` caps at the same time, or
(b) builds a parallel meter pathway and leaves the dead guard for a separate cleanup.
This is a decision point for Youssef (see §B, item 6).

### 5.4 AI-specific limits

Zero. No per-org AI rate limit, no token bucket, no sliding window. The AI controller
has no throttler at all. Bull queue concurrency is set globally
(`celery --concurrency=3` per CLAUDE.md, line 380-388), not per-org.

### 5.5 Account lockout (adjacent but separate concern)

`users.failed_login_attempts` + `users.locked_until` columns. Enforced in
`auth.service.ts:388-451`. 5 failures → 30 min lockout, time-based release.
`SuspiciousLoginService` (Phase 3.3) is a heuristic flagger, not a locker — uses
`user_sessions` query, not a Redis counter. **Not a metering substrate.** Mentioned
only to rule it out.

### 5.6 Bull queue retry config

[notifications.module.ts](backend/src/modules/notifications/notifications.module.ts),
[risk-analysis.module.ts:52](backend/src/modules/risk-analysis/risk-analysis.module.ts),
[obligations.module.ts:36](backend/src/modules/obligations/obligations.module.ts),
[portfolio-export.module.ts:50](backend/src/modules/portfolio-export/portfolio-export.module.ts),
[compliance.module.ts:62](backend/src/modules/compliance/compliance.module.ts),
[email-queue (admin-security.module.ts:57)](backend/src/modules/admin-security/admin-security.module.ts) —
all register queues with NO `defaultJobOptions`. The Bull library default is 1 attempt
(no retry on throw). `learned-baseline.processor.ts:56-120` documents this explicitly:
"default retry policy = 1 attempt." A failed job lands in Bull's failed set; nothing
auto-requeues.

This shapes reconcile: a metered job that throws after reserve cannot count on Bull to
retry. The reserve→release path must be defensive.

### 8.2 Canonical transaction idiom — Bucket 1's establish-identity

The exact pattern the metering reserve should follow lives at
[guest-invitation.service.ts:260-365](backend/src/modules/guest-portal/services/guest-invitation.service.ts):

```typescript
const result = await this.dataSource.transaction(async (manager) => {
  const invitationRepo = manager.getRepository(GuestInvitation);
  const userRepo       = manager.getRepository(User);
  const accessRepo     = manager.getRepository(GuestContractAccess);

  // 1. SELECT FOR UPDATE on the row that gates the transition.
  const invitation = await invitationRepo
    .createQueryBuilder('inv')
    .setLock('pessimistic_write')
    .where('inv.id = :id', { id: invitationId })
    .getOne();
  if (!invitation) throw new UnauthorizedException(...);

  // 2. Defensive re-check under the lock (revoke / expiry).
  if (invitation.status === GuestInvitationStatus.REVOKED ||
      invitation.revoked_at !== null) throw new UnauthorizedException(...);
  if (!invitation.expires_at ||
      invitation.expires_at.getTime() <= Date.now()) throw new UnauthorizedException(...);

  // 3. Race guard: if the user+binding already exists, verify the password
  //    and return the existing row (idempotent retry semantics).
  const existingUser = await userRepo.findOne({
    where: { email: invitation.invited_email, account_type: AccountType.GUEST }
  });
  if (existingUser) {
    const existingBinding = await accessRepo.findOne({
      where: { user_id: existingUser.id, contract_id: invitation.contract_id }
    });
    if (existingBinding) {
      const passwordOk = await bcrypt.compare(dto.password, existingUser.password_hash);
      if (!passwordOk) throw new UnauthorizedException('Invalid password');
      return { user: existingUser, contract_id: invitation.contract_id, wasExisting: true };
    }
  }

  // 4. Insert + flip status.
  const guestUser = userRepo.create({...});
  const savedUser = await userRepo.save(guestUser);
  const binding   = accessRepo.create({...});
  await accessRepo.save(binding);
  invitation.status = GuestInvitationStatus.ACCEPTED;
  await invitationRepo.save(invitation);

  return { user: savedUser, contract_id: invitation.contract_id, wasExisting: false };
});
// post-commit side effects (JWT issuance) happen OUTSIDE the transaction.
```

Shape to imitate for the meter reserve: **(a) pre-flight verify outside txn → (b)
`dataSource.transaction()` → (c) `setLock('pessimistic_write')` on the gating row →
(d) defensive re-check under the lock → (e) write-side mutations → (f) commit → (g)
post-commit side effects.** The 7.18 audit's expectation that "the metering reserve
should follow the canonical pattern" maps directly onto this code.

## 9. Idempotency substrate

### 9.1 Existing patterns by precedent

**Pattern A — single-use via nonce on a DB column.**
[obligation-token.service.ts](backend/src/modules/compliance/services/obligation-token.service.ts):
```typescript
interface MarkMetTokenPayload {
  obligation_id: string;
  user_id: string;
  expires_at: number;
  nonce: string;     // single-use enforcer
}
```
- HMAC-signed token; the `nonce` is also stored on the obligation row.
- Verify: HMAC compare → then check nonce against DB → clear nonce on consumption.
- Result: the second use of the same token sees nonce already cleared → reject.

**Pattern B — reusable inside TTL, no nonce.**
[portfolio-export-token.service.ts:44-152](backend/src/modules/portfolio-export/services/portfolio-export-token.service.ts):
- HMAC-signed token with `expires_at` claim.
- Verification order (regression-tested invariant — see Phase 7.17 lesson #141):
  format → HMAC compare → payload parse → payload expiry claim → DB existence + status
  + user_id match → DB-side expires_at re-check.
- No revocation mechanism; reuse inside TTL is allowed.

**Pattern C — idempotent application-level operations.**
- `revokeInvitation()` ([guest-invitation.service.ts:121-150](backend/src/modules/guest-portal/services/guest-invitation.service.ts)) —
  re-revoke returns the existing row, no error.
- `exchange()` ([guest-invitation.service.ts:172-180](backend/src/modules/guest-portal/services/guest-invitation.service.ts)) —
  stamps `accepted_at` on first call; subsequent exchanges leave it unchanged.
- `establishIdentity()` race guard (§8.2 step 3) — if `(user, contract)` binding
  already exists, verify password and return existing user. Same result, no 409 on
  retry.

### 9.2 The HTTP `Idempotency-Key` header pattern — NOT used

Grep across `backend/src` for `Idempotency-Key` / `idempotency_key` / `Idempotency` as
header name returns zero matches. The codebase has **no standardized request-level
idempotency-key contract**. The metering ledger, which the design says should carry an
`idempotency_key`, would be introducing this pattern from scratch (no precedent to
align to other than the obligation-nonce shape).

### 9.3 Critical gap — Paymob webhook

[subscriptions.service.ts:380-390](backend/src/modules/subscriptions/subscriptions.service.ts)

## 10. AI-call failure handling

### 10.1 NestJS HTTP layer — bare

[ai.service.ts](backend/src/modules/ai/ai.service.ts) — every method is a single
`firstValueFrom(this.httpService.post(...))` with no try/catch, no retry, no
idempotency-key header. Exceptions propagate to the caller as Axios HttpException.

### 10.2 Caller-side handling

[document-processing.service.ts:130-175](backend/src/modules/document-processing/document-processing.service.ts):

- **Dispatch path:** `try { await aiService.startTextExtraction(...) } catch { doc.processing_status = FAILED; await save(); }`.
- **Poll path:** polls `aiService.getJobStatus(jobId)`. On `result.status === 'failed'`,
  writes `processing_status = FAILED` + `error_message`, persists, exits poll loop.
- **Partial result preserved (Phase 7.25):** when text extraction succeeds but quality
  flags fire, status is parked in `HUMAN_REVIEW_RECOMMENDED` (terminal state) — the
  extracted text is persisted even though clause extraction is skipped. Same partial-
  success precedent the metering reconcile may want to follow.
- **No retry, no idempotency key, no compensating action** at the NestJS layer.
  Re-triggering a failed document means calling `reprocess()`, which clears prior
  state and dispatches fresh.

### 10.3 Bull queue retry behaviour

Verified across all queue registrations (see §5.6): **NO `defaultJobOptions.attempts`,
NO `backoff` config anywhere.** Bull's default = 1 attempt. A processor throw lands
the job in the failed set permanently. Reconcile cannot rely on Bull-level retries.

### 10.4 Celery task behaviour

[ai-backend/app/tasks.py:14-204](ai-backend/app/tasks.py):

```python
celery_app.conf.update(
    result_serializer="json",
    result_expires=3600,                  # 1h — RESULT EVAPORATES
    task_track_started=True,
    worker_max_memory_per_child=500_000,
    task_soft_time_limit=300,
    task_time_limit=600,                  # 10 min hard kill
)
```

Each task is decorated `@celery_app.task(name=..., bind=True)` with **NO** `autoretry_for`, **NO** `max_retries`, **NO**
`retry_backoff`. On exception, the task body's own `try/except` returns
`{"status": "failed", "error": str(e)}` — the NestJS poller reads this and translates
to a DB FAILED status.

**Important wrinkle: `result_expires=3600`.** A Celery task result is purged from
Redis after 1 hour. If a reserve happens, the AI job is dispatched, and the NestJS
service crashes / restarts / never polls within 1 hour, the result is unrecoverable.
The reserve is left dangling forever unless reconcile has its own DB-backed state.

The **clause extractor agent has its own in-task retry** at
[clause_extractor.py:558-612](ai-backend/app/agents/clause_extractor.py) — up to 4
attempts on transient errors (429, 5xx, 529) with 30s × 2^(attempt-1) backoff. This
means a single clause-extract dispatch can incur up to 4× Claude token cost without
NestJS seeing any difference in the job result. **This compounds with §3's
no-usage-extraction problem**: not only does NestJS not see token costs, it cannot
even count attempts.

### 10.5 Synthesis for reconcile

Today's pattern is: dispatch → poll → on FAILED, mark entity status and walk away. No
compensating action, no retry, no idempotency, no rollback of any reserved capacity.
The metering reserve→run→reconcile flow is **introducing** the missing pieces — not
extending an existing pattern. This means the build must:

- Add a release step on the reserve side that fires when `getJobStatus()` returns
  `status === 'failed'`.
- Decide what to do with the 1-hour Celery result TTL window (the reservation can
  outlive observability of the result).
- Decide whether in-task retries in the clause extractor should be observable as
  separate reconcile events (so a 4-attempt extract bills 4× tokens) or as one
  event (the reserve was for ONE clause extract; cost overrun on retries is
  absorbed). This depends on §3's decision about token-vs-run-count.

## B. Open questions / decisions this audit surfaces for Youssef

These are the choices that must be made before any code is written. Each is paired
with the section of this audit where the evidence lives.

1. **What does "no active subscription" resolve to?** (§6.4) — Three options:
   - Zero allowances (deny by default; consistent with the dead `SubscriptionGuard`)
   - A platform default row in `plan_allowance` keyed off `plan_id = NULL`
   - A canonical "free" plan seeded into `subscription_plans` and auto-assigned on
     org-create (the `AccountType.FREE` enum value exists already at user.entity.ts:101
     but is never written today)

2. **What `organization_id` does a guest's User row carry?** (§6.7) — The audit
   could not definitively verify the assignment inside `establishIdentity`. The
   resolver's correctness depends on this. Open the `userRepo.create({...})` body in
   `guest-invitation.service.ts` and confirm; if the answer is "the contract's owning
   org", the (caller + contract) → host org resolver collapses to "just read
   `req.user.organization_id` for guest user rows."

3. **Per-endpoint contract-scope verification on `/api/v1/ai/*` endpoints.** (§4.2)
   — `ai.controller.ts` is JWT-only; it relies on each endpoint's body validating the
   `contract_id` belongs to the caller's accessible set. This was NOT exhaustively
   re-verified for every endpoint and is a security pre-requisite to wrapping those
   sites in metering (you cannot meter what you can't access-control).

4. **Token-true-cost vs run-count for reconcile.** (§3.5) — Recording true cost
   requires modifying every ai-backend agent to capture `message.usage.input_tokens` /
   `message.usage.output_tokens` and pipe it through the Celery task result, then
   through `getJobStatus().result.usage`, into reconcile. Run-count is free but
   coarse — and the clause extractor's in-task retries (§10.4) mean it can be
   meaningfully wrong. Pick one explicitly.

5. **In-task retry observability.** (§10.4) — The clause extractor retries up to 4×
   on transient errors. The reserve was for ONE extract; the actual cost can be 4×
   tokens. Does reconcile observe 1 reconcile event (intent-level) or 4 (call-level)?

6. **`SubscriptionGuard` — extend or parallel?** (§5.3) — The plan-limit guard
   exists, works, and is attached to zero endpoints. The metering build can fold
   itself into `SubscriptionGuard` (e.g., extending the metadata key to also accept a
   `meter_key`) or build a separate `MeterGuard` and leave the cleanup for later.

7. **Subject-allowance overrides vs `plan.features`.** (§6.1) — `subscription_plans.features`
   is unused jsonb. The locked design has a separate `subject_allowance` table for
   per-org overrides. Both are defensible; the question is whether ops-time overrides
   live as side-table rows (auditable, but more tables) or inside the plan's
   `features` jsonb (fewer tables, less auditable).

8. **Celery `result_expires=3600` interaction with reserve.** (§10.4) — If reserve
   succeeds but reconcile never gets to read the job result (NestJS crash, > 1 hour
   poll window), the reserved capacity is lost forever. The reconcile design must
   handle this dangling-reserve case — likely via a sweeper that ages-out
   `RESERVED` ledger rows after some TTL.

9. **Metering enum / table naming convention (§12.4) — confirm post-lesson-#143
   naming.** Specifically: `meter_definitions_meter_key_enum` (verbose) vs the older
   short style (`meter_key`). Lessons #31/#103/#111/#143 favour the verbose style;
   this is a one-line confirmation but worth locking before the migration is
   written.

10. **Idempotency-key origin.** (§9.2) — The codebase has no `Idempotency-Key`
    header pattern. The metering ledger's `idempotency_key` column will be the first
    site to use one. Decisions: client-supplied vs server-generated; how it interacts
    with the obligation-nonce precedent (Pattern A in §9.1); whether the meter
    ledger's unique constraint is `(subject_ref, meter_key, idempotency_key)` or
    just `(idempotency_key)`.

## C. What blocks / complicates the locked design

These are the substrate-level frictions surfaced by tasks 3, 5, 6, 8 specifically.

1. **No usage extraction anywhere** (§3.2, §3.4). The locked design's "ledger
   amount" field cannot represent true token cost until every ai-backend agent and the
   embedding service capture `usage` and propagate it through the Celery result. This
   is an ai-backend change touching 9+ agent files. The build's scope must explicitly
   include this or accept run-count metering for v1.

2. **`SubscriptionGuard` is dead code** (§5.3). The substrate is there to enforce
   `max_*` caps; it is attached nowhere. Any new plan-driven enforcement (which is
   what `plan_allowance` is) sits beside dead code that does the same conceptual
   thing. The choice to extend or replace is non-trivial — `SubscriptionGuard`'s
   error shape (`PLAN_LIMIT_PROJECTS` with `{limit, current}`) IS the natural
   `MeterGuard` error shape; reusing it gives consistency, but also resurrects code
   that no one has exercised in production.

3. **No plan seed; no free tier in code** (§6.4, §6.6). The locked design's
   `plan_allowance.plan_id` FK targets a table with no canonical rows. Either the
   build seeds the canonical plans (and accepts the maintenance burden) or it leaves
   them admin-created (which means a fresh database has zero allowances and the
   reserve denies every call).

4. **Paymob webhook idempotency gap (§9.3) is the precedent for what the codebase
   does NOT solve.** The metering ledger's idempotency story is being designed
   greenfield. There is no internal `Idempotency-Key` HTTP pattern to align with
   (§9.2), no central idempotency-key column template, no shared "the second time
   you see this key, return the first result." The build will set the precedent for
   any subsequent payment-webhook fix.

5. **Postgres locking idiom is `setLock('pessimistic_write')`, used at exactly one
   site (§8.2-§8.4).** The metering reserve will be the second site. Whether the
   pattern generalises cleanly under the projected meter cardinality (one meter row
   per `(subject, meter_key, window_key)` — possibly tens-to-hundreds of rows per
   org, hot-locked under concurrent AI calls) is **not verified by precedent**. The
   guest-identity site locks one invitation row per request; the metering reserve
   locks the active meter row per (subject, meter_key). Under bursty concurrent
   guest activity on the same org, lock contention is plausible. Worth modelling.

6. **Bull retries are off, Celery retries are off, ai-backend in-task retries are
   on (§10.3-§10.4).** The retry behaviour is asymmetric across the call chain. The
   reconcile/release path must reason explicitly about this — a Celery task that
   exhausts in-task retries returns `{status: 'failed'}` and Bull does not requeue;
   a reservation made before that chain runs is left orphaned unless reconcile
   sweeps it.

7. **Caller-shape coverage gap on existing AI endpoints (§4.2).** The audit could
   not exhaustively verify that each `/api/v1/ai/*` endpoint validates `contract_id`
   ownership in its body handler. Wrapping these in metering before that
   verification is hazardous: the meter would correctly decrement the right org's
   allowance only if the contract-scope check upstream is correct. A cross-tenant
   contract read was the security regression that PR #42 (commit `54a3959`) fixed
   on the contract entity; the AI endpoints have not received the same audit pass
   since.

8. **Sparse-DB caveat (§0).** The dev DB has 3 contracts and 0 risk_analyses. We
   cannot observe a real AI call end-to-end. Everything in this audit is read-from-
   code, not observed-in-flight. The metering build MUST add a staging-gate
   verification pass against representative data — for the reserve→run→reconcile
   loop to be trusted at scale, the loop needs to be observed firing under realistic
   concurrency, not assumed correct from code review.
